PM·AI — Plan de divulgación

Cómo llevar el plugin de Project Manager con IA a las computadoras del resto del equipo. Tres pasos, en orden: preparar la máquina, instalar el plugin desde Claude Desktop y cargar las credenciales.

1 Software base por máquina

Requisito previo en cada computadora, antes de instalar el plugin.

  • Claude Desktop instalado y con sesión iniciada.
  • Node.js ≥ 22 y pnpm ≥ 11. Forma más simple: corepack enable (ya trae la versión fijada en el repo).
  • Salida de red a Supabase y al endpoint de embeddings/LLM — si hay proxy o firewall corporativo, pedir a TI que lo confirme antes.
El repositorio no distribuye binarios compilados (dist/ está en .gitignore). Después de instalar el plugin hace falta un build local — ver paso 6 de la sección 2.

2 Compartir el repo e instalar desde Claude Desktop

El marketplace del plugin hoy apunta a una ruta local ("source": "."); para instalarlo en otra máquina primero necesita vivir en una URL de git accesible al equipo.

  1. Publicar este repositorio (privado) en el proveedor de git de la empresa, si aún no está ahí.
  2. Compartir la URL del repo con cada compañero — es solo una URL, no un secreto; puede ir por el canal normal del equipo (Slack, correo, etc.).
  3. En Claude Desktop, cada persona ejecuta: 
    /plugin marketplace add <url-del-repo>
  4. Instalar el plugin desde ese marketplace: 
    /plugin install pm-ai@engine-cx-local
  5. Claude Desktop va a pedir los valores de configuración del plugin (supabase_service_key, embeddings_key, llm_key) — se guardan cifrados en el llavero del sistema, nunca en el repo. Ver sección 3 para de dónde salen esos valores.
  6. Compilar una vez, dentro de la carpeta donde Claude Desktop dejó el plugin instalado: 
    pnpm install && pnpm build
    Este paso genera packages/mcp/dist/index.js y los demás binarios que el plugin necesita para arrancar.
  7. Reiniciar Claude Desktop (o recargar plugins) para que tome el servidor MCP nuevo.
  8. Verificar pidiéndole a Claude: "lista mis proyectos". Si responde usando pm_proyectos, la instalación quedó bien.

3 Archivo de credenciales

Cada Project Manager necesita las mismas credenciales para que el plugin funcione: la conexión a la base de datos compartida y, si va a usar /pm-prd, el acceso al repo central de PRDs.

Qué contiene el paquete de credenciales (no son secretos personales, son del equipo):

  • supabase_service_key — acceso de servicio a la base compartida.
  • embeddings_key — obligatoria, para generar los embeddings del índice.
  • llm_key — opcional; sin ella, las descripciones de código usan un fallback sin LLM.
  • ENGINECX_PRD_REPO — ubicación del repo central de PRDs (solo si la persona va a usar /pm-prd). No se comparten credenciales de GitHub: prd-sync publica con la identidad y credenciales de git ya configuradas en el equipo (gh/credential helper/SSH), que /instalar verifica.

Flujo por Project Manager:

  1. Recibe el paquete de credenciales desde el canal seguro designado (gestor de contraseñas del equipo — no texto plano por chat o correo).
  2. Le pasa el archivo/valores a Claude Desktop y le pide: "configura mis credenciales de PM·AI con esto".
  3. Claude completa el userConfig del plugin si falta algún valor, y coloca el .env en la raíz del repo local (nunca dentro de manager/, nunca lo commitea — ya está en .gitignore).
  4. Le pide a Claude que pruebe la configuración: correr /pm-init sobre un repo de prueba, o simplemente pedirle que liste los proyectos existentes.
Antes de repartir esto a todo el equipo: la supabase_service_key da acceso total a la base de todos los proyectos indexados (no solo los propios), y el token de enginecx_prd puede escribir en el repo central de PRDs. Son credenciales de equipo, no de una persona — definan quién puede rotarlas, cuándo, y qué pasa si alguien deja la empresa.

✓ Al terminar

Cada Project Manager queda con el plugin instalado y probado, listo para correr /pm-init la primera vez que arranque con uno de sus proyectos.
